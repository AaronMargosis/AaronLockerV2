AaronLocker’s Office macro control code treats groups of GPOs, often from different GP editor paths, as cohesive units. Overall there are lots of different GP paths, settings, and choices, and corresponding registry keys, value names, and value data. I wrote some code-generation automation to ensure that all the data is captured accurately and repeatably.

Steps:

·        Install the latest Office ADMX files on a workstation. (Download from
https://www.microsoft.com/en-us/download/details.aspx?id=49030 )

·        For each group of settings to treat as a unit, I configure the desired settings by themselves in local GPO, export/back them up using Microsoft’s LGPO.exe (with /b), then convert them to a .PolicyRules file using Microsoft’s Policy Analyzer.

·        In Policy Analyzer, select all the policy rule sets and click View/Compare.

·        In the Policy Viewer window, ensure that none of the Options are enabled.

·        In the Policy Viewer window, Export | Export all data to Excel.

·        Made some manual changes to the Excel: changed some of the header text, changed the values in the first column to true/false, deleted the Policy Type column, ran a macro to change all the CR-LF instances to “||”, changed some text (e.g., moved the ASR rule names from the “Option” columns to the setting name column, and changed the remaining Options text in those columns. (Probably a few other changes too.)

·        Save the resulting .xlsx to a tab-delimited .txt.

·        Run MitigationsAllToCpp.ps1, capturing output to clip.exe (captures to clipboard).

·        Paste into OfficeMacroGpoData.cpp. Cut the extern declarations from the bottom and paste them into OfficeMacroGpoData.h.

This directory contains the *.PolicyRules files I used, the edited Mitigations-All.xlsx and the tab-delimited Mitigations-All.txt that was the input file for MitigationsAllToCpp.ps1 script.
